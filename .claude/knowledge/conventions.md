# Coding Conventions

## File & Folder Naming

- All files use **kebab-case**: `account-worker.processor.ts`, `domain-glossary.md`
- NestJS files follow the pattern: `<name>.<type>.ts` (e.g., `crawler.service.ts`, `crawler.module.ts`)
- Worker processors: `<name>-worker.processor.ts`
- Queue name constants go in a `processors.ts` file per worker module

## Module Structure

Each NestJS module lives in its own directory under `apps/core/src/modules/`:

```
modules/
  <feature>/
    <feature>.module.ts        # Module definition
    <feature>.service.ts       # Business logic
    <feature>.controller.ts    # HTTP endpoints (if needed)
    <feature>.types.ts         # Interfaces and types
```

Worker modules follow a similar pattern under `modules/workers/`:

```
workers/
  <name>-worker/
    <name>-worker.module.ts
    <name>-worker.processor.ts
    processors.ts              # Queue name constants
```

## Standalone Service Modules

Reusable services (e.g., crawler, llm) live as standalone modules under `modules/`:
- Module exports its service via `exports: [ServiceName]`
- Consumers import the module to get access to the service
- Services return **pure data** — no DB writes or storage uploads
- **Processors** (workers) are responsible for persistence: storage uploads, DB upserts, status updates

## Supabase Storage

- Upload files via `supabase.storage.from(bucket).upload(path, content, { upsert: true })`
- Store the storage path (not full URL) in a DB column (e.g., `markdown_url`)
- Download via `supabase.storage.from(bucket).download(path)`
- Bucket names use kebab-case (e.g., `account-websites`)

## Adding a New Worker

1. Create the worker directory under `modules/workers/`
2. Define queue name constant in `processors.ts`
3. Create the processor class extending `WorkerHost`
4. Create the module registering `BullModule.registerQueue()`
5. Import the worker module in `workers.module.ts`

## Import Aliases

- `@echo/supabase-client` — Typed Supabase client (defined in `tsconfig.base.json`)
- All shared libs follow the pattern `@echo/<lib-name>`

## Supabase Client Usage

- Use `@Inject('SUPABASE_CLIENT')` to inject the typed client in any provider
- The client uses the **Service Role Key** (bypasses RLS) — registered globally in `app.module.ts`
- Type: `TypedSupabaseClient` from `@echo/supabase-client`

## Database Migrations

- Files live in `supabase/migrations/` with timestamp prefix: `YYYYMMDDHHMMSS_<description>.sql`
- Use `snake_case` for table and column names
- Always enable RLS on new tables
- Use `int8 PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY` for primary keys
- After migration, run `npm run supabase:types` to regenerate TypeScript types

## Code Style

- **No semicolons** — enforced by Prettier
- **Single quotes** for strings
- **2-space indentation**
- Lint-staged runs ESLint + Prettier on commit for `*.{js,ts,tsx,mts}` files
- TypeScript strict mode enabled (`strict: true` in tsconfig)

## Error Handling

- In worker processors: throw errors to let BullMQ retry (3 attempts, exponential backoff)
- Use `job.log()` for structured logging within workers
- Use NestJS `Logger` for module-level logging

## Config & Environment

- Use `@nestjs/config` `ConfigService` to access env vars in modules
- Raw `process.env` access is acceptable in shared libs (e.g., `supabase-client.ts`)
- Env files: `.env.local` for local dev (gitignored)
